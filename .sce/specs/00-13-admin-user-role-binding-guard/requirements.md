# 00-13 后台账号角色绑定联动校验（Admin User Role Binding Guard）

> 状态：进行中 | 优先级：P0 | 依赖：00-11 platform-admin-console, 00-12 admin-role-permission-tree
> 记录目的：补齐后台账号页在“绑定角色 / 创建账号附带角色”场景下的权限依赖提示与禁用角色回显处理，避免出现静默空列表或提交后端直接拒绝。

## 1. 背景

当前后台账号页已支持：

1. 创建后台账号时附带角色
2. 对已有后台账号执行角色绑定

但当前仍有两个契约缺口：

1. 角色目录查询依赖 `page.system.roles`，若当前账号只有绑定动作权限而无角色页读取权限，页面会出现空角色列表但没有明确解释
2. 后端 `bind-roles` 与 `create user` 的 `roleCodes` 只接受启用角色；若账号已绑定禁用角色，前端直接回填再提交会被后端拒绝

## 2. 范围

### 2.1 本轮必须处理

- 后台账号页的角色目录依赖提示
- 创建账号弹窗的角色可选性说明
- 绑定角色弹窗对“已禁用角色绑定”的显式提示与提交保护
- 前端提交前的角色合法性校验

### 2.2 本轮不处理

- 新增独立的“角色下拉轻量接口”
- 修改后端 `page.system.roles` / `action.system.admin-user.bind-roles` 权限模型
- 自动迁移禁用角色到其他角色

## 3. 需求

### 3.1 角色目录可读性提示

- **R1** 当当前账号没有 `page.system.roles` 时，创建账号与绑定角色场景必须显式提示“无法读取角色目录”
- **R2** 该提示必须说明原因来自后端契约，而不是前端暂无数据
- **R3** 在无法读取角色目录时，角色选择控件必须禁用，不得允许用户在空选项状态下误操作

### 3.2 禁用角色绑定保护

- **R4** 绑定角色弹窗必须识别当前账号已绑定的禁用角色
- **R5** 若存在禁用角色，页面必须明确说明：后端只接受启用角色，本次保存会移除这些禁用角色
- **R6** 前端提交时不得把禁用角色编码继续提交给后端
- **R7** 若当前账号未选择任何启用角色，仍允许提交空数组，表示清空绑定

### 3.3 一致性

- **R8** 创建账号与绑定角色必须共用同一套“可选角色 = 启用角色”规则
- **R9** 提示文案必须围绕真实后端契约，不得把权限不足误描述为接口异常
- **R10** 变更后仍保持后端请求结构为 `roleCodes[] + reason?`

## 4. 验收标准

- [ ] 已新增独立 Spec 并登记增量索引
- [ ] 无角色页读取权限时，页面会明确提示并禁用角色选择
- [ ] 已绑定禁用角色的账号进入绑定弹窗时，会明确提示保存影响
- [ ] 提交时不会继续带出禁用角色编码
- [ ] `npm run type-check` 与 `npm run build` 通过
