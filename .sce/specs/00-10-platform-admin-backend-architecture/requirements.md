# 00-10 平台后台与服务端架构（Platform Admin & Backend Architecture）

> 状态：方案基线已确认 | 优先级：P0 | 适用范围：平台后台、后端服务、数据库设计、会员支付与退款、邀请资格、模板配置
> 记录目的：沉淀本轮关于“后台管理、数据库表、订单退款、会员配置、邀请资格与页面装修配置”的架构结论，作为后续实现与拆分的统一基线。

## 1. 功能概述

当前系统不再是“演员小程序 + 零散接口”，而是升级为完整的平台业务系统：

1. 演员端继续使用微信小程序
2. 平台运营通过独立 Web 管理后台完成审核、配置、订单、退款、资格发放
3. 后端统一采用单体服务承载业务能力
4. 数据层以 MySQL 为事实源，Redis 仅做缓存与幂等辅助

## 2. 当前架构判断

### 2.1 已确认结论

- **R1** 当前阶段不采用微服务，统一使用“模块化单体”架构
- **R2** 小程序端与平台后台共用同一个 Spring Boot 服务和同一个 MySQL
- **R3** 小程序接口统一走 `/api/**`，平台后台接口统一走 `/api/admin/**`
- **R4** 平台后台必须是独立 Web 管理端，不放入 uni-app 小程序仓库中混合开发
- **R5** 小程序只服务演员端用户，运营、审核、模板配置、订单与退款必须进入后台管理端

### 2.2 当前必须补齐的后台能力

- **R6** 必须提供实名认证审核后台
- **R7** 必须提供邀请异常审核与邀请资格管理后台
- **R8** 必须提供会员产品、会员权益、会员状态管理后台
- **R9** 必须提供支付订单与退款管理后台
- **R10** 必须提供名片模板、分享主题、分享产物配置后台

## 3. 产品边界

### 3.1 本轮范围

- 平台后台模块划分
- 服务端模块划分
- 数据库分域与主表规划
- 会员支付订单 / 退款模型
- 邀请规则 / 邀请资格 / 手工下发资格模型
- 模板与页面装修配置的结构化方案

### 3.2 不在本轮范围

- 微服务拆分
- 通用低代码拖拽式装修平台
- 多租户后台
- 复杂 BI 报表平台
- 第三方支付与退款的最终渠道实现细节

## 4. 架构要求

### 4.1 系统形态

- **R11** 系统形态必须为：
  - 演员小程序
  - 平台运营后台 Web
  - 统一 Spring Boot 服务
  - MySQL + Redis
- **R12** 后台 Web 与小程序前端应分仓或至少分工程维护，不与 `kaipai-frontend` 混放
- **R13** 后端按业务域分模块，不允许把后台接口散落到无边界的大 Controller 中

### 4.2 业务域拆分

- **R14** 服务端至少拆分以下业务域：
  - `auth`
  - `actor`
  - `verify`
  - `referral`
  - `membership`
  - `payment`
  - `refund`
  - `card`
  - `fortune`
  - `personalization`
- **R15** `personalization` 只能作为聚合层，负责输出统一读模型，不作为事实数据主表来源

### 4.3 数据建模原则

- **R16** 数据表必须区分：
  - 主数据表
  - 过程表
  - 配置表
  - 日志 / 审计表
- **R17** 所有新表默认复用现有 `BaseEntity` 公共字段规范，不另起一套审计字段体系
- **R18** 聚合结果如 `PersonalizationProfile` 默认不单独落表，优先通过服务聚合生成
- **R19** 冗余字段仅允许用于高频读取、低频变更的聚合结果，如有效邀请数

## 5. 后台模块需求

### 5.1 实名认证后台

- **R20** 后台必须支持查看待审核认证申请
- **R21** 后台必须支持审核通过、审核拒绝、填写拒绝原因
- **R22** 后台审核通过后，必须同步更新用户实名状态与演员认证状态

### 5.2 邀请与资格后台

- **R23** 后台必须支持查看裂变邀请记录
- **R24** 后台必须支持查看异常邀请并进行人工复核
- **R25** 后台必须支持配置邀请规则，如实名前置、资料完整度门槛、同设备限制、小时频率限制
- **R26** 后台必须支持资格与权益的手工下发，不得把“是否有资格”硬编码为单一用户字段

### 5.3 会员后台

- **R27** 后台必须支持会员产品配置，如会员层级、时长、价格、权益说明
- **R28** 后台必须支持查看用户当前会员状态、手工开通、手工延期、手工关闭
- **R29** 会员能力必须与邀请等级解耦，会员与等级分别建模

### 5.4 订单与退款后台

- **R30** 现有 `cooperation_order` 仅用于合作业务，不得复用为会员支付订单
- **R31** 必须新建支付订单模型承载会员购买 / 续费
- **R32** 必须新建退款单模型承载退款申请、审核与渠道退款状态
- **R33** 退款审核状态与退款执行状态必须分离，禁止只用一个状态字段混用

### 5.5 模板与页面配置后台

- **R34** 后台必须支持场景模板、主题 token、分享卡片 / 海报配置的结构化维护
- **R35** 当前阶段不做通用拖拽式页面装修平台，优先做结构化模板配置后台
- **R36** 模板配置、会员能力、邀请规则、资格规则优先通过后台配置和后端下发，不在前端硬编码完整规则库

## 6. 核心数据域要求

### 6.1 保留并扩展现有表

- **R37** 保留现有 `user`、`actor_profile`、`cooperation_order`
- **R38** `user` 可扩展邀请关系、有效邀请计数、注册设备指纹等字段
- **R39** `actor_profile` 可扩展命理相关基础字段，如出生时辰

### 6.2 必须新增的数据表

- **R40** 实名认证域至少新增：
  - `identity_verification`
- **R41** 裂变邀请域至少新增：
  - `invite_code`
  - `referral_record`
  - `referral_policy`
- **R42** 会员域至少新增：
  - `membership_product`
  - `membership_account`
  - `membership_change_log`
- **R43** 交易域至少新增：
  - `payment_order`
  - `payment_transaction`
  - `refund_order`
  - `refund_operate_log`
- **R44** 页面与模板域至少新增：
  - `card_scene_template`
  - `actor_card_config`
  - `actor_share_preference`
- **R45** 命理域至少新增：
  - `fortune_report`
- **R46** 权益与资格域至少新增：
  - `user_entitlement_grant`
  - `entitlement_rule`
- **R47** 后台治理域至少新增：
  - `admin_user`
  - `admin_role`
  - `admin_user_role`
  - `admin_operation_log`
  - `template_publish_log`

### 6.3 禁止复用和禁止混用

- **R48** 现有 `invite_record` 仅代表剧组邀约，不得复用于裂变邀请
- **R49** 合作订单、支付订单、退款单必须分表，禁止混在同一张订单表中
- **R50** 资格、权益、会员状态不得仅依赖单一布尔字段表达

## 7. 安全与审计要求

- **R51** 身份证号必须加密存储，并保留可用于去重的摘要字段
- **R52** 后台查看敏感数据时，必须有权限控制和操作审计
- **R53** 所有审核、退款、资格发放、模板发布操作必须记录操作日志
- **R54** Redis 不得作为业务真相来源，只能用于缓存、限流、幂等和短期风控辅助

## 8. 验收标准

- [ ] 已形成后台 Web + 小程序 + 单体服务的统一架构结论
- [ ] 已明确后台最小模块边界：审核 / 邀请 / 会员 / 订单 / 退款 / 配置 / 权限
- [ ] 已明确合作订单、支付订单、退款单三类订单边界
- [ ] 已明确邀请裂变与剧组邀约模型分离
- [ ] 已明确会员等级、邀请等级、资格权益三套模型分离
- [ ] 已明确模板配置采用结构化后台，而非低代码拖拽装修
- [ ] 已形成后续数据库 DDL 与后台开发的统一基线
