# 00-27 小程序前端架构总览（Mini Program Frontend Architecture）

> 状态：架构基线已建立 | 优先级：P0 | 适用范围：`kaipai-frontend` 小程序前端整体结构、页面信息架构、分层边界、分包治理、共享体系、当前主线治理
> 记录目的：补齐当前仓库缺失的“小程序前端整体架构总纲”，把分散在 00-01 / 00-02 / 00-03 / 00-05 / 05-05 / 05-11 中的前端架构事实收敛成统一入口，作为后续新增页面、重构、分包、组件抽取和主线治理的最高级前端架构基线。

## 1. 功能概述

当前仓库已经具备：

1. 样式体系与页面骨架
2. `Kp` 共享组件体系
3. `api / utils / stores / types` 基础分层
4. 小程序分包和包体审计机制
5. 当前名片分享与个性化主线

但缺少一份面向 `kaipai-frontend` 的总览式前端架构文档，导致：

- 新任务只能从局部页面 Spec 理解结构，难以把握整体边界
- “页面放主包还是分包”“逻辑放页面还是 utils”“哪些 UI 必须抽组件”缺少统一判断基线
- 文档层面存在“有专题约束，但没有总纲”的断层

本 Spec 的目标不是替代各专题 Spec，而是为整个小程序前端提供统一的系统边界、信息架构和决策规则。

## 2. 当前架构判断

### 2.1 已确认结论

- **R1** `kaipai-frontend` 必须明确定位为演员端微信小程序前端工程。
- **R2** 当前小程序产品主线必须明确为“演员名片分享 + 命理驱动个性化 + 会员能力分层 + 统一分享产物”。
- **R3** 平台后台 Web 不属于本工程，必须继续由独立工程 `kaipai-admin` 承担。
- **R4** 当前前端已不是“单一主包 + 零散页面”结构，而是“主包 + `pkg-card` + `pkg-tools`”结构。
- **R5** `05-11 fortune-driven-share-personalization` 是当前演员增强主线的前端治理基线。

### 2.2 当前必须补齐的内容

- **R6** 必须补齐一份前端整体架构总纲，作为所有局部 Spec 的上位入口。
- **R7** 必须明确页面信息架构，包括主包、分包、页面类型和当前主线入口。
- **R8** 必须明确目录职责、共享层边界、数据层边界、解析层边界和包体决策规则。
- **R9** 必须明确“哪些内容允许在页面内实现，哪些必须抽为共享组件 / Store / utils”。

## 3. 产品边界

### 3.1 本 Spec 覆盖范围

- 小程序前端仓库定位
- 页面信息架构与页面分层
- 目录结构与职责边界
- 共享组件、样式系统、状态/API/类型层边界
- 分包策略与 2 MB 包体治理约束
- 当前业务主线在前端中的落位方式
- 新增需求进入实现前的前端决策规则

### 3.2 本 Spec 不覆盖范围

- 单个页面的具体交互细节
- 单个业务功能的详细状态流转
- 后端接口实现、数据库设计
- 平台后台 Web 的信息架构
- AI/LLM 后端封装策略细节

## 4. 总体前端架构要求

### 4.1 系统形态

- **R10** 小程序前端必须继续使用 uni-app + Vue 3 + TypeScript。
- **R11** 小程序前端只承载演员端主链路、名片分享链路和分享落地链路，不承载后台运营功能。
- **R12** 小程序前端与平台后台前端必须保持双工程结构，不得重新混放。

### 4.2 当前主线定位

- **R13** 当前前端主线必须围绕“演员档案 -> 名片 -> 分享产物 -> 个性化主题 -> 会员分层 -> 邀请成长”组织。
- **R14** 信用积分 / 排行榜只能作为历史方案保留，不得重新进入当前前端总纲。
- **R15** 当前名片分享产物必须包含小程序分享卡片、海报、公开名片页和邀请卡片，不再只把“名片页”本身视为全部分享链路。

## 5. 页面信息架构要求

### 5.1 页面层分组

- **R16** 页面层必须按以下三组管理：
  - `pages/`：主包基础页与 tab 页
  - `pkg-card/`：演员增强主线页
  - `pkg-tools/`：独立工具页
- **R17** 新页面进入工程前，必须先回答“它属于哪一组页面”。

### 5.2 当前主包页面分类

- **R18** `pages/` 必须继续承载：
  - 登录与角色选择
  - 首页与我的页
  - 角色详情、投递确认、我的投递等基础演员链路
  - 档案编辑、公开详情等基础资料页
- **R19** 主包页面默认代表启动即达页、tab 页或高频基础链路页。

### 5.3 当前 `pkg-card` 分包分类

- **R20** `pkg-card/` 必须继续承载：
  - 名片主预览页
  - 会员能力中心
  - 实名认证
  - 邀请裂变
  - 命理个性化说明与应用页
- **R21** `pkg-card/` 的定位必须明确为“演员增强主线分包”，而不是临时放工具页的杂项分包。

### 5.4 当前 `pkg-tools` 分包分类

- **R22** `pkg-tools/` 必须继续承载低频独立工具页，如 `webview`、`video-player`。
- **R23** 工具页若不直接参与演员增强主线，应优先放入 `pkg-tools/`，避免污染 `pkg-card/`。

### 5.5 页面类型

- **R24** 页面顶部策略必须继续服从 `SHARED_CONVENTIONS.md` 的页面类型分类，不允许每个页面重新定义顶部实现方式。
- **R25** 页面开发前必须明确是：
  - 深色 Hero 页
  - 普通顶部页
  - Tab 根页

## 6. 目录与职责边界要求

### 6.1 页面目录

- **R26** 页面层只负责视图组装、生命周期、页面级交互流转。
- **R27** 页面层不得直接承担整套业务规则编排、重复接口拼装和重复 UI 壳层实现。

### 6.2 共享组件目录

- **R28** 所有跨页面复用 UI 必须放入 `src/components/Kp*.vue`。
- **R29** 页面中出现两次及以上、且未来可扩散的 UI 片段，必须优先评估抽为 `Kp` 组件。
- **R30** 返回按钮壳层、固定底部操作栏、邀请摘要卡、能力矩阵、等级卡、主题预览等均属于优先共享对象。

### 6.3 API / Store / Type / Utils 目录

- **R31** 接口调用只能通过 `src/api/*.ts` 暴露。
- **R32** 跨页面共享状态必须优先进入 `src/stores/*.ts`。
- **R33** 共享领域类型必须统一进入 `src/types/*.ts`。
- **R34** 解析、格式化、校验、主题解析、分享产物解析等复用逻辑必须统一进入 `src/utils/*.ts`。

## 7. 共享体系要求

### 7.1 共享 UI 体系

- **R35** 当前前端共享 UI 体系必须以 `Kp` 组件为唯一前缀标准。
- **R36** 页面内不允许复制相同的按钮组、信息卡壳层、邀请摘要壳层、顶部返回壳层。
- **R37** 新增共享组件时，必须回填 `00-02 shared-components`。

### 7.2 样式体系

- **R38** 所有页面必须服从 `00-01 global-style-system` 的 token 和 mixin 体系。
- **R39** 所有页面必须服从 `00-08 style-system-governance` 和 `00-09 shared-style-shells` 的共享样式壳层规则。
- **R40** 页面不得为了局部需求脱离 token 和 mixin 体系自造一套样式语言。

## 8. 数据与状态架构要求

### 8.1 API 层

- **R41** 页面不得直接拼接原始请求。
- **R42** API 模块应按业务域组织，不得按“页面一页一个 API 文件”无限扩散。
- **R43** 当前主线至少应继续保留 `auth / actor / level / invite / verify / fortune / personalization` 等业务域边界。

### 8.2 Store 层

- **R44** 登录态、角色态、实名状态、等级信息、会员态、邀请统计等共享信息必须优先由 Store 承担。
- **R45** 只有确实跨页面共享、且有持续状态需求时，才新增 Store；不能把一次性页面状态滥放到全局 Store。

### 8.3 类型层

- **R46** 页面和组件不得私下复制相同 DTO/VO 结构。
- **R47** 个性化、邀请、认证、等级、用户、档案等共享领域模型必须统一由类型层定义。

## 9. 解析层要求

### 9.1 解析层定位

- **R48** `utils/` 中的解析器是当前主线的关键中间层，负责把后端数据、用户态、模板配置、命理输入组合为页面可消费结果。
- **R49** 页面必须消费解析结果，而不是重复拼接主题、分享路径、能力 gating 和展示文案。

### 9.2 当前主线解析基线

- **R50** 当前名片主线的个性化、主题、分享产物解析必须继续围绕：
  - `personalization`
  - `theme-resolver`
  - `share-artifact`
- **R51** 后续相关重构不得把这些统一逻辑重新散落回页面。

## 10. 分包与包体治理要求

### 10.1 包体约束

- **R52** 小程序单包不得超过 2 MB。
- **R53** 分包治理不是可选优化项，而是前端架构约束的一部分。

### 10.2 新增页面决策规则

- **R54** 新页面进入实现前，必须先判断是否应放入分包。
- **R55** 若页面具备以下任一特征，应优先进入分包：
  - 非 tab 页
  - 独立入口
  - 持续增长的资源或逻辑
  - 非启动必经链路
- **R56** 登录、首页、我的 tab、启动即达链路默认保留主包。

### 10.3 分包角色边界

- **R57** `pkg-card` 承担演员增强主线，不得塞入无关工具页。
- **R58** `pkg-tools` 承担独立工具页，不得演变为第二个主线业务分包。

## 11. 当前主线与历史边界要求

- **R59** `05-05` 继续承接名片分享和等级/会员基础能力，但后续结构调整必须服从 `05-11`。
- **R60** `05-08` 的命理页面定位必须是“个性化输入源说明与应用页”，而不是孤立功能岛。
- **R61** `05-10` 的邀请页必须从“裂变页面”升级为统一分享产物体系的一环。
- **R62** 历史信用主线不得重新进入当前前端信息架构。

## 12. 开发决策规则

- **R63** 新需求进入实现前，必须先回答以下问题：
  - 页面放主包还是分包
  - UI 是否已有共享组件可复用
  - 业务规则应放页面、Store、API 还是 utils
  - 是否需要更新 00-27、00-02、00-05 或 05-11
- **R64** 若某项改动影响前端整体分层或页面放置策略，必须优先更新 00-27，再更新局部 Spec。
- **R65** 若某项改动会引入跨页面重复 UI，必须先抽共享组件再继续扩展页面。

## 13. 与其他 Spec 的关系

- **R67** 00-01 / 00-02 / 00-03 / 00-05 / 00-08 / 00-09 负责前端基础设施专题。
- **R68** 01-xx / 02-xx / 03-xx / 04-xx / 05-xx 负责页面与业务专题。
- **R69** 局部 Spec 若与 00-27 冲突，以 00-27 先修正文档边界，再决定实现归属。
