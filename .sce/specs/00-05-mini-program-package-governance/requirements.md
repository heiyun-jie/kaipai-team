# 00-05 小程序包体治理（Mini Program Package Governance）

> 状态：进行中 | 优先级：高 | 依赖：00-04, 05-05, 05-07

## 1. 目标

为当前 `mp-weixin` 主线建立明确的小程序包体治理规则、审计能力和迁移边界，确保后续名片、会员、分享相关页面继续演进时，不会失控堆积到单包中。

## 2. 本轮范围

- 微信小程序构建产物包体审计
- 2MB 单包约束基线落地
- 当前页面主包 / 分包候选盘点
- 审计命令与报告沉淀

## 3. 需求

### 3.1 包体约束

- **R1** 当前主线必须显式记录“单包体积不超过 2MB”的治理约束。
- **R2** 包体治理不能只停留在文档，必须提供可执行的本地审计能力。
- **R3** 审计结果必须能区分主包与各分包体积；若当前尚未分包，也必须能输出“全部仍在主包”的结果。

### 3.2 审计能力

- **R4** 必须提供针对 `dist/build/mp-weixin` 的包体审计脚本。
- **R5** 审计脚本必须读取构建后的 `app.json`，根据 `subPackages` / `subpackages` 真实配置计算包体，而不是手写猜测。
- **R6** 当任一包超过阈值时，审计脚本必须返回失败状态，便于后续纳入发布前检查。

### 3.3 当前主线盘点

- **R7** 必须记录当前仍在主包的页面清单，并识别后续优先迁移候选。
- **R8** 候选迁移方案必须优先考虑非 tab 页、工具页、会员页、分享页和后台残留页，不得先动登录、首页、我的 tab 等主入口。
- **R9** 本轮允许只完成治理基线和迁移方案，不强制在同一轮完成所有页面分包迁移。

### 3.4 后续开发默认策略

- **R10** 后续新增功能模块时，默认必须先评估是否应以独立分包形式承载，而不是直接进入主包。
- **R11** 若模块具备明确入口、非 tab、可独立访问、资源或脚本可能持续增长等特征，应优先按独立分包设计。
- **R12** 只有在启动即达、主链路必经、tab 页面直接依赖等场景下，才允许默认放入主包。

### 3.4 重构边界

- **R13** 本轮不得因为包体治理而改坏现有页面跳转语义。
- **R14** 本轮若未实施真实分包迁移，必须给出明确的后续迁移顺序和验证方式。

## 4. 验收标准

- [ ] 已新增小程序包体治理 Spec
- [ ] 已提供 `mp-weixin` 构建产物包体审计脚本
- [ ] 已形成当前主包体积基线报告
- [ ] 已给出后续分包迁移优先级建议
