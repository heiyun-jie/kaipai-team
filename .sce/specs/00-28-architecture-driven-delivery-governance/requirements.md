# 00-28 架构驱动的项目推进治理（Architecture-Driven Delivery Governance）

> 状态：规划中 | 优先级：P0 | 依赖：00-10, 00-11, 00-27, 05-11
> 记录目的：建立一套“以整体架构为源头推进多个工程开发”的统一治理方法，把当前项目从“按页面 / 按人 / 按临时需求”推进，收敛为“按架构边界 / 业务能力 / 跨工程闭环”推进。

## 1. 功能概述

当前仓库已经有：

1. 平台后台与服务端架构基线 `00-10`
2. 平台后台管理端基线 `00-11`
3. 小程序前端架构总纲 `00-27`
4. 当前演员增强主线治理基线 `05-11`

但仍缺少一份“如何根据整体架构去推动多个项目开发”的交付治理总纲，导致：

- 需求容易被拆成零散页面任务，而不是业务闭环
- 前端、后台、后端、数据的推进顺序容易脱节
- 某些功能先做了页面，却没有上游数据、权限、状态和验收口径
- 缺少一个统一判断“现在先推什么、后推什么、谁和谁必须同轮闭环”的基线

本 Spec 的目标不是替代业务 Spec，而是建立一套更上位的推进规则。

## 2. 当前治理判断

### 2.1 已确认结论

- **R1** 整体架构必须成为项目推进的第一入口，不再由页面列表或临时需求主导开发顺序。
- **R2** 当前项目推进的最小交付单元必须是“业务能力闭环”，而不是单独一个页面或单独一个接口。
- **R3** 业务能力闭环默认至少要回答四个交付面：
  - 数据 / 状态
  - 后端接口 / 规则
  - 后台治理能力
  - 小程序用户侧落点
- **R4** 某项功能若缺少上游数据模型、后台治理入口或状态回写路径，不得只在前端页面层伪完成。

### 2.2 当前必须补齐的治理内容

- **R5** 必须明确当前阶段有哪些“项目级工作流”，以及它们之间的依赖顺序。
- **R6** 必须明确从整体架构到具体开发任务的拆分方法。
- **R7** 必须明确跨工程交付的统一验收口径。
- **R8** 必须明确哪些项目属于当前主线，哪些只是历史保留或次级优化。

## 3. 覆盖范围

### 3.1 本 Spec 覆盖范围

- 项目推进的拆分规则
- 架构到工作流的映射关系
- 当前阶段的优先级顺序
- 跨工程闭环的验收要求
- 项目推进时的节奏、入口和收口方式

### 3.2 本 Spec 不覆盖范围

- 单个页面的详细交互
- 单个接口的字段定义
- 某个团队的具体人员分工
- 日常排期工具和任务系统的具体使用方式

## 4. 推进单元要求

### 4.1 最小推进单元

- **R9** 推进单元必须优先定义为“能力切片（Capability Slice）”。
- **R10** 一个能力切片至少包含：
  - 业务目标
  - 依赖的上位架构 Spec
  - 涉及的数据模型
  - 涉及的后端接口
  - 涉及的后台页面或治理动作
  - 涉及的小程序页面或前端能力
  - 验收方式
- **R11** 不允许把“页面 UI 已出”视为切片完成。

### 4.2 切片命名原则

- **R12** 切片命名必须使用业务能力语义，如“实名认证闭环”“邀请资格闭环”“名片分享产物闭环”，而不是“某页改版”。
- **R13** UI 优化、样式对齐、包体治理、权限树等基础设施型工作，必须作为基础治理切片单独管理，不混入业务闭环切片。

## 5. 当前项目工作流要求

### 5.1 工作流分组

- **R14** 当前阶段项目推进至少按以下四组工作流管理：
  - 全局治理与共享基础
  - 平台治理与服务端核心域
  - 小程序演员主线与分享主线
  - AI 能力与个性化增强

### 5.2 全局治理与共享基础

- **R15** 该工作流负责：
  - Spec 与文档治理
  - 共享组件 / 类型 / API / utils / 样式基线
  - 分包治理与前端架构总纲
  - 跨工程约定的收敛

### 5.3 平台治理与服务端核心域

- **R16** 该工作流负责：
  - 数据表与状态模型
  - 后端服务域边界
  - 后台治理页面与权限
  - 实名、邀请、会员、订单、退款、模板配置等平台能力

### 5.4 小程序演员主线与分享主线

- **R17** 该工作流负责：
  - 演员档案
  - 名片分享
  - 公开详情
  - 会员能力前台承接
  - 邀请裂变前台承接
  - 命理个性化前台承接

### 5.5 AI 能力与个性化增强

- **R18** 该工作流负责：
  - AI 文案优化
  - AI 简历润色
  - 个性化输入源聚合
  - 分享产物级能力控制
- **R19** AI 工作流不得脱离现有业务主线孤立推进，必须挂接到真实页面和真实业务闭环。

## 6. 当前阶段优先级要求

### 6.1 优先级判断规则

- **R20** 优先级必须先看“是否为上游使能能力”，再看“是否为当前主线能力”，最后才看“是否为体验增强项”。
- **R21** 若某项能力是另一条主线的前置条件，应优先于下游体验页。
- **R22** 历史保留页面、非主线功能和孤立优化不得抢占主线能力的实现优先级。

### 6.2 当前建议优先级

- **R23** P0：全局治理与平台基础能力
  - `00-10` 平台后台与服务端架构
  - `00-11` 平台后台管理端
  - `00-27` 小程序前端架构总纲
  - `05-09` 实名认证
  - `05-10` 邀请裂变
- **R24** P1：当前演员增强与分享主线
  - `05-11` 命理驱动的分享定制主线
  - `05-05` 名片分享与会员能力
  - `05-08` 命理个性化
- **R25** P2：AI 业务增强
  - `05-04` AI 简历润色
- **R26** `05-04` 不应早于实名认证、邀请、会员、名片分享主线去推动真实交付。

## 7. 架构到开发任务的拆分要求

### 7.1 拆分顺序

- **R27** 每次推进一个能力切片时，必须按以下顺序拆分：
  1. 上位架构基线
  2. 数据与状态模型
  3. 后端接口与规则
  4. 后台治理入口
  5. 小程序或前端用户侧落点
  6. 联调与验收

### 7.2 禁止的拆分方式

- **R28** 禁止直接从页面原型反向倒推全部实现，而不先确认数据和状态边界。
- **R29** 禁止把后台治理能力遗漏后，只做演员端页面。
- **R30** 禁止只做接口清单，不定义状态回写、权限和审计。

## 8. 验收要求

### 8.1 统一完成定义

- **R31** 一个能力切片完成时，至少需满足：
  - 对应 Spec 已落地或已更新
  - 数据模型和状态流转清楚
  - 后端接口可追溯
  - 后台治理动作可操作
  - 前台 / 小程序落点可验证
  - 文档、类型、验证记录同步回填

### 8.2 跨工程闭环要求

- **R32** 对于实名、邀请、会员、模板、分享产物、AI 等核心能力，验收必须同时覆盖：
  - 后端
  - 后台 Web
  - 小程序前端
- **R33** 若只有其中一端落地，默认视为“局部完成”，不得标记为整项完成。

## 9. 当前主线与历史边界要求

- **R34** 当前项目推进必须以 `05-11` 为演员增强主线治理基线。
- **R35** `04-xx` 剧组端小程序页面当前属于历史保留，不应再作为当前主线扩展重心。
- **R36** `05-03 credit-score` 仅作历史保留，不应重新占用当前主线排期。

## 10. 验收标准

- [ ] 已建立一份以整体架构为入口的项目推进治理 Spec
- [ ] 已明确当前阶段的工作流分组与优先级顺序
- [ ] 已明确“能力切片”而不是“单页面任务”作为最小推进单元
- [ ] 已明确跨工程闭环的统一验收口径
- [ ] 后续推动开发时，可以直接用本 Spec 判断“先做什么、一起做什么、什么算完成”
