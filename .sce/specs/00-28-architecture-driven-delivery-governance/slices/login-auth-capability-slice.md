# 登录鉴权与前台会话闭环切片卡

## 1. 切片名称

登录鉴权与前台会话闭环

## 2. 切片目标

把小程序登录链路从“手机号 / 微信按钮可演示、前端可 mock”推进到“后端真实鉴权、邀请码透传、会话恢复、受保护数据请求时序和运行配置显式可治理”的最小闭环。

本切片的目标不是单做一个登录页，而是建立：

- 手机号验证码登录 / 注册的真实后端契约
- 微信手机号一键登录的真实后端契约与显式能力开关
- 登录成功后的会话恢复与前台受保护数据请求时序
- `inviteCode` 在短信注册与微信自动注册中的统一透传
- 运行时配置缺失时的显式降级，而不是前端假装成功

## 4. 业务范围

### 4.1 本轮范围

- 小程序手机号验证码发送、登录、注册
- 小程序微信手机号一键登录
- 登录态恢复、`/api/user/me`、身份切换与演员侧运行态同步
- 邀请码在登录页、注册接口、微信自动注册中的透传
- 微信能力的前端显式开关与后端配置校验
- 运行时缺配置时的前台文案降级与联调台账

### 4.2 不在本轮范围

- 后台账号登录、SSO、RBAC 重构
- 短信网关正式商用接入
- 微信开放平台 UnionId 体系和多端账号融合
- 第三方风控账号体系

## 5. 数据与状态模型

### 5.1 关键实体 / 表 / 缓存

- `user`
- `sms:code:{phone}` Redis 键
- JWT Token
- `invite_code`
- `referral_record`

### 5.2 关键状态

#### 登录态

- 未登录
- 短信验证码已发送
- 已登录
- 已注册待选身份

#### 微信登录可用态

- mock 演示态
- 后端接口已接通但前端未显式启用
- 前后端均启用
- 配置缺失阻塞

### 5.3 状态流转

```text
未登录
  -> 发送短信验证码
  -> 手机号登录 / 注册
  -> 返回 token + user session
  -> 建立前台会话
  -> 再同步 verify / invite / level 等受保护数据

未登录
  -> 微信 getPhoneNumber 返回 code
  -> /auth/wechat-login
  -> 已注册用户直接登录 or 新用户自动注册
  -> 若携带 inviteCode，则在自动注册事务内消费
  -> 返回 token + user session

若缺微信 appId / appSecret 或前端未显式启用
  -> 页面只展示“稍后接入”提示
  -> 不允许继续走前端 mock 成功链
```

## 6. 后端交付

### 6.1 核心接口

#### 小程序端

- `POST /api/auth/sendCode`
- `POST /api/auth/login`
- `POST /api/auth/register`
- `POST /api/auth/wechat-login`
- `GET /api/user/me`
- `PUT /api/user/role`

### 6.2 核心服务规则

- 登录成功后的唯一事实源必须是后端返回的 token + user session，不允许页面自己拼会话
- 页面首屏如果依赖前台 token，必须先建立会话，再请求受保护数据
- `inviteCode` 只能在首次注册或微信自动注册时消费，登录时不得伪造绑定
- 微信登录若缺 `appId / appSecret`，后端必须显式报错，不能伪造成功
- 前端若未显式启用微信登录能力，只能展示降级提示，不能继续走 mock
- `user.me`、`level.info`、`invite.stats` 等登录后摘要必须围绕同一用户事实回填

### 6.3 安全 / 权限 / 审计

- 验证码必须带时效与错误校验
- 登录、注册、身份切换都必须返回稳定 DTO
- 自动注册与邀请绑定必须在同一事务边界内处理
- 运行时配置缺失必须作为外部依赖阻塞记录到联调结论，不能口头忽略

## 7. 后台 / 运行配置交付

### 7.1 管理动作

- 维护小程序登录运行时配置台账
- 确认前端 `VITE_API_BASE_URL`、`VITE_USE_MOCK`、`VITE_ENABLE_WECHAT_AUTH`
- 确认后端 `WECHAT_MINIAPP_APP_ID`、`WECHAT_MINIAPP_APP_SECRET`
- 记录真实环境联调账号、错误提示和阻塞项

### 7.2 本轮说明

- 本轮不新增独立“登录治理后台页面”
- 后台交付收口为运行配置台账、联调准入条件和错误可观测口径
- 若后续需要独立配置中心或审计页，应在本切片基础上再拆子卡

## 8. 小程序 / 前台交付

### 8.1 页面落点

- `pages/login/index`
- `stores/user`
- `utils/runtime`
- `utils/request`

### 8.2 前端 gating / 展示 / 回写

- 登录页同时承接手机号验证码与微信手机号授权
- 登录页承接 `inviteCode / scene`，并在注册或微信自动注册时透传
- 建立会话后再同步 `verify / invite / level`
- 微信登录按钮显隐由显式运行时开关控制
- 若未启用或配置缺失，页面只展示“稍后接入”，不再用 mock 掩盖

## 9. 联调点

- 手机号发送验证码、注册、登录、会话恢复链路稳定
- 微信登录在真实环境能区分“老用户登录”和“新用户自动注册”
- `inviteCode` 能穿过短信注册与微信自动注册
- 登录完成后 `user.me / invite / verify / level` 按正确时序同步
- 配置缺失时的前台提示与后端报错口径一致

## 10. 当前阻塞项

- 当前阶段高优先级阻塞已清空：手机号登录 / 注册、`inviteCode` 透传、会话恢复与页面证据已完成真实样本回填
- 微信真实环境依赖 `WECHAT_MINIAPP_APP_ID / SECRET` 与前端显式启用开关，当前已由 `00-48` 明确降级为后续能力批次
- 正式短信网关商用接入已由 `00-51` 明确降级为后续能力批次，不再阻塞当前阶段闭环判断

## 11. 建议推进顺序

1. 锁定登录与会话事实源
2. 固化后端接口与 DTO
3. 固化运行时显式开关与配置台账
4. 回接前端会话恢复与登录后同步
5. 真实环境联调并回填结论

### 11.1 第一步：锁定事实源

- 锁定 token + `user.me` 为会话事实源
- 锁定 `inviteCode` 只在注册事务内消费
- 锁定微信登录配置缺失时的阻塞口径

### 11.2 第二步：固化后端

- 打通短信登录 / 注册
- 打通微信手机号登录
- 打通 `user.me` 与身份切换

### 11.3 第三步：固化运行时开关

- 前端显式控制微信登录可见性
- 后端显式校验微信配置
- 不再允许真实链路兜回 mock 成功

### 11.4 第四步：回接前端

- 登录页、`stores/user`、请求层共同承接登录态
- 登录成功后再同步受保护数据
- 邀请码和会话恢复保持单一路径

### 11.5 第五步：联调与验收

- 跑通手机号登录 / 注册
- 验证邀请码透传和登录后状态同步
- 当前阶段页面证据要补齐 `login -> mine -> membership -> invite`
- 若未来明确推进微信 / 正式短信批次，再补微信老用户登录 / 新用户自动注册与正式短信样本

## 12. 完成定义

### 12.1 局部完成

以下只能算局部完成：

- 登录页 UI 可点，但后端仍是 mock
- 微信按钮可见，但未显式校验运行时配置
- 会话建立后没有统一同步受保护数据
- `inviteCode` 只在短信注册生效，微信自动注册丢失

### 12.2 当前阶段闭环完成

以下同时满足，才能算当前阶段闭环完成：

- 手机号验证码登录 / 注册走真实后端契约
- 登录成功后能稳定建立会话并同步受保护数据
- `inviteCode` 在短信注册链路中口径一致，且能通过后台记录回读
- `login(带inviteCode) -> mine -> membership -> invite` 页面级证据已补齐
- 配置缺失时能显式阻塞并给出一致提示，不再伪造成功
- `00-28` 已回填真实环境联调结论与阻塞项

### 12.3 全量闭环完成

以下同时满足，才能算未来全量闭环完成：

- 当前阶段闭环条件已全部满足
- 微信手机号一键登录走真实后端契约，并由显式运行时开关控制
- `inviteCode` 在微信自动注册中口径一致
- 正式短信网关已商用接入，并具备真实送达 / 失败治理口径
