# AI 简历润色闭环切片卡

## 1. 切片名称

AI 简历润色闭环

## 2. 切片目标

在演员档案编辑页建立一个可控的 AI 协作闭环：基于全档案上下文和用户指令生成字段级 patch，支持预览、按字段应用、撤销、历史回滚，并把 AI 能力纳入等级 / 配额 / 安全约束体系。

## 4. 业务范围

### 4.1 本轮范围

- 编辑页 AI 润色入口
- 全档案上下文组装
- 多轮对话
- 结构化 patch 返回
- 预览、按字段应用、整批应用、撤销、历史回滚
- 配额、超时、敏感词、失败兜底

### 4.2 不在本轮范围

- 前台直连大模型
- 自动无确认覆盖表单
- 图像 / 视频内容理解
- 高级知识库、RAG、多模型编排

## 5. 数据与状态模型

### 5.1 关键实体 / 表

- 演员档案文本字段聚合结果
- AI 对话消息状态
- 字段级 patch 结果
- AI 使用配额（现有 `AiUsageQuota` 口径）
- 润色历史快照（建议后续独立持久化）

### 5.2 关键状态

- 对话状态：
  - 空闲
  - 发送中
  - 已返回 patch
  - 失败 / 超时
- patch 状态：
  - 未应用
  - 已按字段应用
  - 已整批应用
  - 已撤销

### 5.3 状态流转

```text
演员打开编辑页
  -> 进入 AI 面板
  -> 发送自然语言指令
  -> 后端 / mock 返回字段级 patch
  -> 前端展示修改前后对比
  -> 用户按字段应用 / 整批应用
  -> 写回表单
  -> 形成历史快照
  -> 可撤销 / 回滚
```

## 6. 后端交付

### 6.1 核心接口

- 已有前端口径：
  - `GET /api/ai/quota?type=resume_polish`
  - `POST /api/ai/polish-resume`
- 后续建议补齐：
  - `GET /api/ai/resume-polish/history`
  - `POST /api/ai/resume-polish/history/{id}/rollback`
  - `PUT /api/actor/profile` 扩展 `aiResumeApplyMeta`

### 6.2 核心服务规则

- 所有 AI 调用统一由后端封装，前端不直连模型
- 输入上下文必须是整份档案，不是单字段孤立文本
- 输出必须是可解析的字段级 patch，而不是自由文本
- patch 只代表草稿，不代表档案已保存
- 演员档案真实写库仍统一走 `PUT /api/actor/profile`
- 结构化事实字段不能被静默改写
- 超时、配额、敏感内容和不可解析响应都必须有兜底

### 6.3 安全 / 权限 / 审计

- 系统 prompt 必须显式禁止虚构经历、品牌、奖项
- 用户输入需经过基础敏感词过滤
- AI 调用与写回动作建议纳入操作 / 审计记录
- 配额扣减必须以服务端为准，不能只靠前端计数

## 7. 后台交付

### 7.1 管理页 / 治理动作

本轮不要求单独上线后台 AI 运维页，但平台侧至少需要预留：

- AI 配额策略配置入口
- 风险内容或失败样本回看能力
- 后续 AI 调用成本 / 失败率监控入口

### 7.2 运营侧关键动作

- 调整不同等级或会员态的 AI 配额
- 回看异常 patch、失败调用和敏感内容命中
- 在规则变更时同步调整前台文案和 gating

## 8. 小程序 / 前台交付

### 8.1 页面落点

- `pages/actor-profile/edit`
- 关联验证页：
  - `pkg-card/actor-card/index`
  - `pages/actor-profile/detail`

### 8.2 前端 gating / 展示 / 回写

- AI 入口不替代原手动编辑
- 配额、认证状态、等级能力要在前端统一展示
- AI 返回结构必须按字段展示 diff
- 用户确认前不得覆盖表单
- 应用结果需要联动到名片摘要和公开详情展示验证

## 9. 联调点

- 编辑页 AI patch 应用后，名片页和公开详情页要能消费到最新文本
- 认证、等级、AI 配额展示必须和等级中心一致
- 失败 / 超时 / 敏感内容的提示文案要统一

## 10. 当前阻塞项

- 后端还没有真实 AI / LLM 封装
- `05-04` 当前只有 requirements/tasks，没有已落地前后端协议
- 需要固定字段级 patch 结构，否则前端无法稳定预览和应用
- 需要确认 `draftId -> actor/profile save -> history` 的固化链路

## 11. 建议推进顺序

1. 数据与状态
   - 明确全档案上下文结构、消息模型、patch 模型、历史快照模型
2. 后端规则与接口
   - 先定义结构化响应协议
   - 先走 mock / adapter 路径，再替换为真实模型封装
3. 前台回接
   - 在编辑页挂 AI 面板、diff 预览、字段应用、撤销和历史
   - 与 `stores/user.ts` 的配额 / 认证 / 等级态对齐
4. 联动验证
   - 把应用结果同步验证到名片页和公开详情页
5. 真实模型接入
   - 用后端统一封装替换 mock，接入配额扣减、超时和敏感词策略

## 12. 完成定义

### 12.1 局部完成

- 只有 AI 按钮，没有真实上下文和 patch 协议
- 只有 mock 润色结果，没有预览确认和撤销
- 只有编辑页写回，没有名片页 / 公开页联动验证

### 12.2 闭环完成

- 编辑页具备 AI 入口、多轮对话和字段级 patch 预览
- 用户可按字段应用、整批应用、撤销和历史回滚
- AI 配额、认证前置、失败兜底、安全约束全部生效
- 应用结果联动到名片页和公开详情页
- 后端统一封装真实模型，前端不直连大模型
