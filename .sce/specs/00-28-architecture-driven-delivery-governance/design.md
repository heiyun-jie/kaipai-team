# 架构驱动的项目推进治理 - 技术设计

_Requirements: 00-28 全部_

## 1. 设计目标

00-28 解决的不是某一个功能，而是“如何把当前整体架构转成可执行的项目推进方式”。

本 Spec 主要回答五个问题：

1. 应该按什么单位推进开发
2. 哪些工作流可以并行，哪些必须前后依赖
3. 当前阶段先推哪些项目
4. 一个项目要怎样从架构拆到具体实现
5. 什么才算真正完成一项能力

## 2. 总体推进模型

```text
整体架构基线
  -> 工作流分组
  -> 能力切片
  -> 跨工程交付项
  -> 联调与验收
  -> 文档回填
```

### 2.1 整体架构输入

当前项目推进的上位输入来自：

- `00-10 platform-admin-backend-architecture`
- `00-11 platform-admin-console`
- `00-27 mini-program-frontend-architecture`
- `05-11 fortune-driven-share-personalization`

明确结论：

- `00-10` 定义平台后台、服务端、数据域和接口边界
- `00-11` 定义后台 Web 的治理动作、菜单、权限和页面落点
- `00-27` 定义小程序前端的信息架构、分层和分包边界
- `05-11` 定义当前演员增强和分享主线

00-28 的职责是把这些架构结论转成推进方法。

## 3. 工作流设计

### 3.1 并行执行总则

- `00-28` 允许按“基础治理 / 平台核心域 / 小程序主线 / AI 增强”四条工作流推进
- 各工作流允许并行，但必须通过能力切片在里程碑点统一收口
- 页面、接口、数据和配置的推进顺序不得脱离上位架构基线

## 4. 能力切片模板

### 4.1 标准结构

每个推进项目都先整理成一个能力切片，至少回答：

| 字段 | 说明 |
|------|------|
| 切片名称 | 用业务能力命名 |
| 上位 Spec | 依赖的架构或业务 Spec |
| 业务目标 | 解决什么问题 |
| 数据模型 | 依赖哪些表 / 状态 / 索引 |
| 后端交付 | 哪些接口 / 服务规则 |
| 后台交付 | 哪些运营动作 / 管理页 |
| 小程序交付 | 哪些页面 / 前端能力 |
| 联调点 | 哪些地方必须同轮闭环 |
| 验收口径 | 什么叫完成 |

### 4.2 示例：实名认证闭环

| 字段 | 内容 |
|------|------|
| 切片名称 | 实名认证闭环 |
| 上位 Spec | `00-10`、`00-11`、`05-09` |
| 业务目标 | 演员提交实名信息，平台审核，状态同步回写并影响会员 / 邀请等前置条件 |
| 数据模型 | `identity_verification`、用户实名状态、审核状态、拒绝原因、审计字段 |
| 后端交付 | 提交接口、状态查询接口、后台列表 / 详情 / 审核接口、状态回写逻辑 |
| 后台交付 | 待审核列表、历史列表、详情抽屉、通过 / 拒绝操作、操作日志 |
| 小程序交付 | 提交页、状态展示、名片 / 会员页联动拦截 |
| 联调点 | 审核通过后，小程序状态同步；审核拒绝后，前台原因可见；后台动作进入日志 |
| 验收口径 | 后台可审、前台可见、状态可回写、日志可追溯 |

结论：

- 这类能力不能拆成“前台实名页”和“后台审核页”两项孤立任务
- 正确拆法应该是一个实名认证闭环切片，内部再拆数据、后端、后台、小程序子任务

## 5. 当前阶段优先级路线

### 5.1 第一阶段：平台基础能力先行

先推进：

1. 实名认证闭环
2. 邀请裂变与邀请资格闭环
3. 会员能力与模板配置闭环
4. 后台账号、角色、权限、日志治理

原因：

- 这些能力是平台事实层和治理层
- 不先稳定这层，前台分享和 AI 能力会失去可控边界

### 5.2 第二阶段：演员增强与分享主线

再推进：

1. 名片预览与分享产物
2. 命理个性化输入源
3. 会员分层前台回接
4. 邀请前台入口与分享联动

原因：

- 这层消费平台能力和配置
- 属于当前产品主线，但不应抢跑上游治理能力

### 5.3 第三阶段：AI 业务增强

最后推进：

1. `05-04 ai-resume-polish`
2. 分享文案级 AI 优化
3. 更深的个性化推荐或生成式能力

原因：

- AI 能力对上下文、权限、会员 gating、状态和回滚要求更高
- 它应该建立在主线业务稳定后再落

## 7. 完成定义

### 7.1 局部完成

以下情况只能算局部完成：

- 页面已出，但接口未稳定
- 接口已出，但状态回写未打通
- 后台页已出，但操作日志未接
- 小程序功能已出，但没有对应后台治理入口

### 7.2 闭环完成

以下条件同时满足，才能算闭环完成：

1. 上位 Spec 已存在并对齐
2. 数据模型、接口、状态流转清楚
3. 后台治理入口可操作
4. 小程序或前台用户侧落点可验证
5. 关键日志、权限、限额或回滚约束已接入
6. 文档、映射表、验证记录已回填

## 8. 使用方式

后续你推动开发时，不再直接说“先做哪个页面”，而是先回答：

1. 这件事属于哪个工作流？
2. 它对应哪个能力切片？
3. 这个切片的数据、后端、后台、小程序分别是什么？
4. 它的上游前提有没有准备好？
5. 这一轮的验收是局部完成还是闭环完成？

如果这 5 个问题答不出来，说明当前还不适合直接排开发。
