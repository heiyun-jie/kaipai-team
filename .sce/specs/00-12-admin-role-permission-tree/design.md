# 后台角色权限树形编排 - 技术设计

_Requirements: 00-12 全部_

## 1. 设计目标

在不修改后端接口契约的前提下，把角色权限编辑从三段权限码多选升级为结构化树形编排，降低配置错误率并提升可审计性。

## 2. 设计约束

1. 后端仍以 `AdminRoleSavePayload` 的 `menuPermissions`、`pagePermissions`、`actionPermissions` 为唯一提交模型
2. 权限码仍以后端 / 现有前端常量为准，不新增虚构权限
3. 新交互必须优先复用现有后台常量和菜单模型，避免再次平行维护一套页面语义

## 3. 权限 registry

新增统一权限 registry，至少包含：

- 模块 key
- 模块标签
- 权限类别：`menu` / `page` / `action`
- 权限码
- 展示标签

模块分组优先复用 `src/constants/menus.ts` 的一级业务域；未直接出现在侧边菜单中的权限页或操作权限，按所属业务域补齐到 registry。

## 4. 树结构

树节点采用三层结构：

1. 业务模块
2. 权限类别分组（菜单权限 / 页面权限 / 操作权限）
3. 权限叶子节点

约束：

- 只有叶子节点对应真实权限码
- 父节点只承担组织结构，不直接写回后端
- 勾选父节点时，自动勾选其下全部叶子权限

## 5. 前端映射策略

树形组件内部只维护“已勾选叶子权限码集合”。

提交前执行分类拆分：

- `menu.*` -> `menuPermissions`
- `page.*` -> `pagePermissions`
- `action.*` -> `actionPermissions`

回显时把三数组合并为统一勾选集合，回填到树组件。

## 6. 页面改造点

### 6.1 角色编辑弹窗

- 把三段 `el-select multiple` 替换为树形权限编辑器
- 顶部提供关键字过滤、已选统计、清空操作
- 保留角色编码、角色名称、状态、备注字段

### 6.2 角色详情抽屉

- 权限标签按模块或类别展示业务名称
- 标签中可保留原始权限码作为辅助信息，但不再只显示 code

## 7. 验证方式

1. 创建角色并勾选多模块权限，确认提交 payload 仍为三数组
2. 编辑已有角色，确认原权限正确回显到树
3. 复制角色后重新打开详情，确认权限数量与内容不丢失
4. 运行 `npm run type-check`
5. 运行 `npm run build`
