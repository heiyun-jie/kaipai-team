# 00-12 后台角色权限树形编排（Admin Role Permission Tree）

> 状态：进行中 | 优先级：P0 | 依赖：00-11 platform-admin-console
> 记录目的：补齐后台角色管理中“权限编辑仍是原始权限码多选”的功能缺口，定义树形权限编排的页面形态、数据边界和验收标准。

## 1. 背景

`00-11` 已交付后台角色管理页，但当前角色编辑仍采用菜单权限、页面权限、操作权限三段原始权限码多选。

现状问题：

1. 权限项只展示原始 code，不利于运营和管理员理解模块边界
2. 当前允许自由输入权限码，存在录入非标准权限码的风险
3. 菜单、页面、操作之间缺少结构化视图，角色授权难以审计

## 2. 范围

### 2.1 本轮必须处理

- 角色新建 / 编辑页的权限录入方式
- 权限项的模块化分组与树形展示
- 权限选择结果与后端 `menuPermissions`、`pagePermissions`、`actionPermissions` 三数组的映射
- 角色详情页的权限展示可读性

### 2.2 本轮不处理

- 后端角色权限模型变更
- 新增后台权限码或重命名现有权限码
- 通用 RBAC 可视化设计器
- 基于数据库动态下发权限矩阵

## 3. 需求

### 3.1 权限编排形态

- **R1** 角色编辑必须提供树形权限编排，不再使用三段原始权限码多选作为主交互
- **R2** 权限树必须按后台业务模块分组，至少覆盖工作台、实名认证、邀请裂变、会员中心、订单中心、退款中心、页面配置、系统管理
- **R3** 每个业务模块下必须继续区分菜单权限、页面权限、操作权限
- **R4** 权限树必须支持展开 / 收起、清空选择和关键字过滤

### 3.2 权限数据边界

- **R5** 前端提交给后端的结构必须继续保持 `menuPermissions`、`pagePermissions`、`actionPermissions` 三个数组
- **R6** 权限树只能选择系统内已登记的标准权限码，不得允许自由输入未注册权限码
- **R7** 权限树中的展示名称必须使用业务语义，不直接把原始权限码作为唯一可读信息
- **R8** 未进入权限矩阵 registry 的权限码必须以显式兜底文案展示，不得静默丢失

### 3.3 页面一致性

- **R9** 角色详情页必须能按可读标签展示已分配权限
- **R10** 列表中的权限概览统计逻辑必须与树形选择结果一致
- **R11** 复制角色时必须保留原角色权限集合，不因前端视图切换而丢失

## 4. 验收标准

- [ ] 角色编辑已从原始多选升级为树形权限编排
- [ ] 前端提交仍保持后端既有三数组契约
- [ ] 已禁止自由输入未登记权限码
- [ ] 角色详情与编辑页都能以业务语义展示权限
