### 我的原始理解

#### `kaipai-frontend`

小程序做前端展示。

1. 登录注册 - 完善档案
2. 首页展示 - 档案列表
3. 个人中心 - 用户展示
4. 个人 - 分享裂变
5. 个人 - 名片
6. 实名认证
7. 会员展示

#### `kaipai-admin`

后台管理

1. 用户角色划分
2. 订单体系
3. 会员体系
4. 支付 / 退款体系
5. 实名认证
6. 分享链路
7. 装修页面

#### `kaipaile-server`

1. 数据库 + redis
2. 前后端接口（区分）
3. 表 / 字段

### 收敛后的正式结论

- `kaipai-frontend`：演员端微信小程序，负责登录、档案编辑、首页业务浏览、名片分享、实名认证提交、会员能力展示等用户前台交互。
- `kaipai-admin`：平台运营后台 Web，负责审核、会员、订单、退款、邀请裂变、模板配置、账号角色权限、操作日志等后台治理能力。
- `kaipaile-server`：统一的 Spring Boot 业务服务，分别对小程序提供 `/api/**`，对后台提供 `/api/admin/**`，负责业务规则、数据读写、权限控制、状态流转和服务聚合。
- `MySQL`：事实数据源，承载表、字段、状态和审计记录。
- `Redis`：缓存、幂等、限流、短期辅助，不是业务真相来源。

### 基于 `00-27` 补充：小程序前端架构

来源：

- `.sce/specs/00-27-mini-program-frontend-architecture/requirements.md`
- `.sce/specs/00-27-mini-program-frontend-architecture/design.md`

#### 1. 小程序前端的系统定位

`kaipai-frontend` 不是“放页面的前端目录”，而是独立的小程序前端工程。

它当前只服务三条线：

1. 演员基础链路
2. 演员增强主线
3. 分享链路

不负责：

1. 平台后台运营
2. 后端业务规则
3. 数据库存储与审计

#### 2. 小程序前端的页面分组

当前不是“单主包 + 零散页面”，而是三层页面架构：

```text
kaipai-frontend
  -> pages/**      主包基础页与 tab 页
  -> pkg-card/**   演员增强主线分包
  -> pkg-tools/**  工具分包
```

对应职责：

- `pages/`：登录、角色选择、首页、我的页、角色详情、投递、档案编辑、公开详情等基础链路
- `pkg-card/`：名片、会员、实名认证、邀请裂变、命理个性化等演员增强主线
- `pkg-tools/`：`webview`、`video-player` 等独立工具页

#### 3. 小程序前端的模块分层

小程序前端不能只按“页面 + 接口”理解，还要按分层理解：

```text
页面层         pages/**, pkg-card/**, pkg-tools/**
共享 UI 层     src/components/Kp*.vue
状态层         src/stores/*.ts
API 层         src/api/*.ts
类型层         src/types/*.ts
解析/工具层    src/utils/*.ts
样式层         src/styles/*
```

各层职责：

- 页面层：负责视图组装、生命周期、页面交互
- 共享 UI 层：负责 `Kp` 组件复用
- 状态层：负责跨页面共享状态
- API 层：负责按业务域组织接口调用
- 类型层：负责共享领域类型
- 解析/工具层：负责格式化、校验、主题解析、分享产物解析、个性化聚合
- 样式层：负责 token、mixin、页面骨架

#### 4. 当前小程序主线

当前小程序前端主线不是旧的信用积分，而是：

```text
演员档案
  -> 名片
  -> 分享产物
  -> 个性化主题
  -> 会员分层
  -> 邀请成长
```

这意味着：

1. `05-11` 是当前前端治理主线
2. `pkg-card` 是当前增长重点分包
3. 新功能不能再按旧的信用主线组织

#### 5. 重要架构判断

1. 小程序前端与后台前端必须继续双工程维护，不能混放。
2. 新页面进入前，必须先判断是主包、`pkg-card` 还是 `pkg-tools`。
3. 重复 UI 不应继续写在页面里，应优先抽成 `Kp` 组件。
4. 共享状态不应散落页面，应进入 Store。
5. 个性化、主题、分享产物等组合逻辑不应散落页面，应进入 `utils/` 解析层。
6. 分包治理不是优化项，而是前端架构约束的一部分。

#### 6. 当前我应该记住的最简版

如果以后快速复习，只记这句：

`kaipai-frontend` 是一个按“主包 + 演员增强分包 + 工具分包”组织的 uni-app 小程序前端，页面负责组装，`Kp` 组件负责共享 UI，`api/store/types/utils/styles` 负责共享基础设施，当前主线是“档案 -> 名片 -> 分享 -> 个性化 -> 会员 -> 邀请”。

### 今天修正的关键点

1. `kaipaile-server` 不是“数据库 + redis”，而是“统一业务服务”。
2. `MySQL` 和 `Redis` 属于数据层依赖，不等于服务端本身。
3. 架构表达不能只写“页面功能”，要写清楚“系统职责边界”。
4. 当前阶段的正确关键词是“模块化单体”，不是微服务。
5. 小程序前端也不能只按“页面列表”理解，要按“页面分组 + 分层边界 + 主线定位”理解。

### 复习时要回答的问题

1. 为什么这个项目当前阶段适合模块化单体，而不是微服务？
2. 为什么后台接口和小程序接口要分成 `/api/**` 和 `/api/admin/**`？
3. 为什么 `Redis` 不能作为业务真相来源？
4. 小程序前端为什么要分成 `pages / pkg-card / pkg-tools`？
