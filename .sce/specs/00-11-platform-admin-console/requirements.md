# 00-11 平台后台管理端（Platform Admin Console）

> 状态：进行中 | 优先级：P0 | 依赖：00-10 platform-admin-backend-architecture
> 记录目的：定义平台运营后台 Web 的模块边界、菜单结构、页面能力、权限模型与实施范围，作为后台前端与后台管理接口的统一产品基线。

## 1. 功能概述

平台后台管理端是运营与管理员使用的独立 Web 应用，用于承接当前小程序主线无法在演员端内完成的审核、配置、订单、退款、资格发放与运营治理能力。

后台管理端本轮核心目标：

1. 承接实名认证人工审核
2. 承接邀请异常复核与邀请资格管理
3. 承接会员产品、会员状态与权益配置
4. 承接会员支付订单与退款处理
5. 承接名片模板、主题 token、分享产物配置
6. 承接后台账号、角色、权限与操作日志

## 2. 当前产品判断

- **R1** 后台管理端必须是独立 Web 应用，不得实现为小程序内页面
- **R2** 后台管理端仅面向平台运营、审核人员、内容配置人员和管理员
- **R3** 后台管理端与小程序共享同一服务端和同一数据库
- **R4** 后台能力范围必须与 00-10 的服务端与数据库架构基线保持一致

## 3. 产品边界

### 3.1 本轮范围

- 后台登录与权限控制
- 后台菜单与信息架构
- 审核类页面
- 配置类页面
- 订单与退款页面
- 会员与资格管理页面
- 操作日志页面

### 3.2 不在本轮范围

- BI 大屏
- 自定义报表设计器
- 通用低代码搭建平台
- 多租户 SaaS 后台
- 审核任务分派工作流引擎

## 4. 一级模块要求

### 4.1 工作台

- **R5** 后台首页必须提供工作台概览
- **R6** 工作台至少展示待审核实名认证数、待处理异常邀请数、待处理退款数、今日支付订单数

### 4.2 用户中心

- **R7** 后台必须支持按用户 ID、手机号、昵称、角色查询用户
- **R8** 后台必须支持查看用户的实名状态、邀请状态、会员状态、资格状态、基础档案信息
- **R9** 后台必须支持查看用户关联的操作轨迹摘要，但不得在用户页直接承担复杂审核流程

### 4.3 实名认证

- **R10** 后台必须提供实名认证待审核列表
- **R11** 后台必须支持查看认证申请详情、脱敏身份信息、提交时间、拒绝历史
- **R12** 后台必须支持审核通过与审核拒绝
- **R13** 审核拒绝时必须填写原因

### 4.4 邀请裂变

- **R14** 后台必须支持查看邀请码、邀请记录、邀请统计
- **R15** 后台必须支持查看异常邀请记录
- **R16** 后台必须支持对异常邀请执行“通过 / 作废 / 标记复核完成”操作
- **R17** 后台必须支持邀请规则配置
- **R18** 后台必须支持邀请资格下发与撤销

### 4.5 会员中心

- **R19** 后台必须支持会员产品配置
- **R20** 后台必须支持会员权益说明配置
- **R21** 后台必须支持查看单个用户当前会员状态和会员变更记录
- **R22** 后台必须支持手工开通会员、延期会员、关闭会员

### 4.6 订单中心

- **R23** 后台必须支持支付订单列表与详情查看
- **R24** 后台必须支持查看支付流水、支付渠道单号、支付状态、关联会员产品
- **R25** 后台不得把合作订单与会员支付订单混放在同一页面语义中

### 4.7 退款中心

- **R26** 后台必须支持退款申请列表
- **R27** 后台必须支持退款审核
- **R28** 后台必须支持查看退款执行状态与渠道退款信息
- **R29** 退款页面必须明确区分“待审核”“退款中”“退款成功”“退款失败”

### 4.8 页面配置

- **R30** 后台必须支持场景模板配置
- **R31** 后台必须支持主题 token 配置
- **R32** 后台必须支持分享卡片与海报配置
- **R33** 后台必须支持模板上下架、排序、版本发布记录
- **R34** 当前阶段页面配置应以结构化表单为主，不做拖拽式装修

### 4.9 系统管理

- **R35** 后台必须支持后台账号管理
- **R36** 后台必须支持角色与权限管理
- **R37** 后台必须支持操作日志检索

## 5. 菜单结构要求

- **R38** 一级菜单至少包括：
  - 工作台
  - 用户中心
  - 实名认证
  - 邀请裂变
  - 会员中心
  - 订单中心
  - 退款中心
  - 页面配置
  - 系统管理
- **R39** 二级菜单应与业务域一一对应，避免一个页面同时承担审核、配置、发放三类职责
- **R40** 菜单命名必须使用业务语义，不得使用开发内部术语

## 6. 权限模型要求

- **R41** 后台至少支持以下角色：
  - `super_admin`
  - `verify_operator`
  - `referral_operator`
  - `membership_operator`
  - `finance_operator`
  - `content_operator`
- **R42** 菜单权限、页面权限、操作权限必须分离
- **R43** 审核、退款、资格发放、模板发布必须走显式操作权限，不得仅凭页面可见即默认可操作
- **R44** 高风险操作必须记录操作日志

## 7. 页面形态要求

- **R45** 列表页统一提供筛选区、表格区、分页区
- **R46** 审核页统一提供详情抽屉或详情页，不在列表表格中塞入过多字段
- **R47** 配置页统一区分“草稿 / 已发布”视角
- **R48** 用户状态、订单状态、退款状态、审核状态必须使用统一状态标签体系

## 8. 接口边界要求

- **R49** 后台接口统一走 `/api/admin/**`
- **R50** 后台页面只消费后台聚合接口，不直接拼装多个前台接口
- **R51** 审核和配置操作应显式区分查询接口与写入接口

## 9. 与现有主线的关系

- **R52** 05-09 的实名认证前端提交页依赖本 Spec 定义的后台审核闭环
- **R53** 05-10 的邀请裂变依赖本 Spec 定义的异常邀请审核与邀请资格管理
- **R54** 05-11 的模板、会员能力、分享产物配置依赖本 Spec 定义的页面配置与会员后台

- [ ] 已明确审核、配置、订单、退款、资格、权限六类核心后台模块
- [ ] 已明确后台角色与权限最小集合
- [ ] 已明确后台接口统一走 `/api/admin/**`
