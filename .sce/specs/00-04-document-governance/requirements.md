# 00-04 文档治理（Document Governance）

> 状态：进行中 | 优先级：高 | 依赖：05-05, steering

## 1. 目标

当前仓库已完成产品主线切换，但文档层仍存在旧信用主线、旧页面结构和旧组件说明。需要建立一轮明确的文档治理，把当前有效信息更新到主文档，把失效但仍需保留的历史信息归档，而不是继续混放在同一入口文档里。

## 2. 范围

### 2.1 本轮必须处理

- `docs/product-design.md`
- `docs/dev-playbook.md`
- `.sce/specs/README.md`
- `.sce/specs/spec-code-mapping.md`
- 必要的归档目录与归档说明

### 2.2 本轮不处理

- 历史 Spec 正文中的旧方案细节
- 运维私有文档中的敏感信息
- 后端接口文档体系重写

## 3. 需求

### 3.1 主文档必须只承载当前主线

- **R1** `docs/product-design.md` 必须改写为当前产品主线文档，主线应明确为“演员名片分享 + 会员分层”。
- **R2** 主文档中不得继续把“信用积分 / 排行榜”写成当前功能范围、当前页面入口或当前验收目标。
- **R3** 主文档中如需提及旧信用方案，只能以“历史方案 / 已搁置”的形式简短说明，并明确已归档。

### 3.2 历史内容必须归档

- **R4** 旧版产品设计文档需要归档到明确路径，保留版本和日期信息。
- **R5** 归档后的主文档必须给出历史文档入口，避免信息直接丢失。
- **R6** 归档文档应视为历史参考，不得继续作为当前实现依据。

### 3.3 文档事实必须与代码基线一致

- **R7** Spec 索引、代码映射、开发手册中的页面数量、类型数量、工具模块数量、主线页面状态必须与当前仓库一致。
- **R8** 开发手册必须补充当前主线的文档规则和历史/现行边界。
- **R9** 当前主线文档中需要明确 `05-05` 为现行主线，`05-03` 为搁置历史方案。

### 3.4 治理结果必须可审计

- **R10** 本轮文档治理需保留治理记录，明确已更新和已归档内容。
- **R11** 治理完成后，非历史层文档中不应再把“我的信用分”“积分记录”“排行榜”“KpCreditBadge”“KpLevelTag”写成当前主线内容。

## 4. 验收标准

- [ ] 已创建文档治理 Spec
- [ ] `docs/product-design.md` 已重写为当前主线文档
- [ ] 旧版产品设计文档已归档并可追溯
- [ ] `docs/dev-playbook.md` 已补充当前主线文档治理约束
- [ ] `.sce/specs/README.md` 已纳入文档治理 Spec
- [ ] `.sce/specs/spec-code-mapping.md` 的页面/类型/工具映射已更新
- [ ] 非历史层文档的旧信用主线表述已清理
