# 05-07 当前主线组件化重构（Mainline Component Refactor）

> 状态：进行中 | 优先级：高 | 依赖：05-02, 05-05, 05-06

## 1. 目标

对当前主线中的重复页面行为进行组件化 / composable 化重构，优先处理重复度高、风险低、可复用范围广的部分，降低页面体积和后续维护成本。

## 2. 本轮范围

- 浮动返回导航定位逻辑
- 媒体选择逻辑（相册 / 拍照 / 视频选择）
- 低风险重复展示组件
- 当前主线及其直接依赖页面中的重复接入

## 3. 需求

### 3.1 浮动返回导航

- **R1** 页面内重复的 `getMenuButtonBoundingClientRect`、`navStyle`、`backButtonStyle` 计算必须抽出复用能力。
- **R2** 抽取后不得改变现有页面返回按钮的位置、尺寸和交互行为。
- **R3** 当前主线至少应覆盖 `actor-card`、`actor-profile/edit` 等核心页面。

### 3.2 媒体选择

- **R4** 图片 / 视频来源选择（相册 / 拍照）逻辑必须抽出复用能力。
- **R5** 抽取后不得改变现有页面的上传前交互和错误处理语义。
- **R6** 当前主线至少应覆盖演员资料编辑页和剧组资料编辑页。

### 3.3 重构边界

- **R7** 本轮以收敛重复代码为目标，不新增业务功能。
- **R8** 不得为了抽象而改坏现有页面 UI 结构。
- **R9** 若某处结构虽相似但语义差异较大，本轮允许暂不抽组件，避免过度抽象。

### 3.4 低风险展示组件

- **R10** 当前主线允许补充低风险纯展示组件，用于收敛重复的段落头、胶囊选择器、菜单项外壳等结构。
- **R11** 新展示组件必须保持现有文案、交互语义和页面层级不变，不得借重构名义改业务入口。
- **R12** 展示组件抽取应优先覆盖 `pages/actor-card/index` 与 `pages/mine/index` 中重复度高的结构。

### 3.5 小程序包体约束

- **R13** 本轮重构必须显式关注微信小程序包体控制，不得因抽象引入明显冗余依赖或重复资源。
- **R14** 涉及页面、资源和复用模块布局时，必须以“单包体积不超过 2MB”为约束进行评估；若后续页面继续增长，应优先规划分包而不是继续堆主包。
