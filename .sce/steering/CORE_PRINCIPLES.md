# 核心开发原则（基准规则）

> 模板只保留长期原则。当前阶段状态写入 `CURRENT_CONTEXT.md`，详细规则写入项目文档。

## 1. 先建 Spec，再动实现

- 所有需求先进入 `.sce/specs/<spec>/`，再进入设计、任务、实现与验证。
- 产物归档到对应 Spec，避免散落根目录。

## 2. Steering 只保留长期有效内容

- `CORE_PRINCIPLES.md` 放长期原则。
- `ENVIRONMENT.md` 放项目级运行与发布规则。
- `CURRENT_CONTEXT.md` 放当前阶段最小上下文。
- `RULES_GUIDE.md` 放职责边界与迁移规则。

## 3. 默认自主闭环推进

- 默认流程：分析 → 修改 → 测试 → 修复 → 文档同步 → 交付。
- 非阻断问题不应频繁等待确认。

## 4. 复用已有机制，不要平行造轮子

- 已有能力优先复用，例如缺陷经验与发布阻断统一走 `errorbook`。
- 若接管项目中已存在自定义缺陷复盘册，必须统一盘点并吸收到 `.sce/errorbook`，不要长期并存混用。
- 不要在 steering 中额外定义另一套错题、发布、会话或治理模式。

## 5. 质量问题必须追根

- 禁止靠跳过测试、关闭校验、吞错回退来伪装成功。
- 临时兜底必须附带退出条件、清理任务与截止时间。
- 重写或替换实现时，同步删除失效旧代码、旧样式、死分支和无效引用。
- 如果因兼容、灰度或回滚必须临时保留旧实现，必须写明保留理由、边界、退出条件和后续清理计划。

## 6. 测试、文档、代码必须同步闭环

- 代码变更必须跑相关验证；发布前不得忽略失败。
- 重要功能、命令、配置变化必须同步更新 README、用户文档或发布说明。
- 功能修改、UI 重写、模块替换时，已失效的旧实现、旧样式、死分支、失效适配层和无效引用必须在同一轮变更中清理，不得继续留作“保险”。
- 若因兼容、灰度或回滚必须暂时保留旧实现，必须明确保留理由、适用边界、退出条件和后续清理计划。

## 7. 业务场景未知时先澄清，禁止直接彻底禁用

- 不了解业务场景、模块、页面、实体或业务约束时，默认动作是先补上下文、缩小范围、生成澄清问题。
- 禁止把“未知业务场景”直接等同于 `deny`、`disable`、`answer-only` 或其他一刀切兜底禁用。
- 只有在明确命中安全、权限、合规或破坏性规则后，才允许进入阻断。
- 这条规则适用于所有接入 SCE 的项目，不允许按项目例外绕过。

## 8. 禁止盲改问题，必须先建立问题契约和证据

- 修改问题前，必须先明确现象、复现条件、影响范围、预期行为和验证方式。
- 缺少日志、数据、接口样本、最小复现或问题契约时，应先补证据，不得靠猜测连续改代码。
- 若两轮修改仍未收敛，必须回到调试、定位和根因分析。

## 9. Steering 条目变更必须先评估，禁止随意增删

- 新增、删除或重写 steering 条目前，必须先评估它是否真属于长期原则，是否与现有条目重复，是否应迁移到 `CURRENT_CONTEXT.md`、Spec 或项目文档。
- steering 变更必须说明触发原因、适用范围以及与现有规则的关系。
- 接管、升级和治理脚本只能补齐基线、修复漂移，不能把未经评估的项目习惯直接塞进 steering。

## 10. 问题修复时前后端接口不一致默认以后端契约为准

- 若前端调用后端 API 出现路径、方法、字段、状态码或响应结构不匹配，默认以后端现有接口契约为准。
- 除非明确要求新建接口或修改后端接口，否则禁止为了迁就前端错误调用去随意改后端。
- 默认优先修正前端请求、映射、类型和兼容处理。

## 11. 单文件规模过大必须触发重构评估

- 若项目尚未建立自己的阈值，默认参考源文件 `2000 / 4000 / 10000` 行三档触发：分别对应“必须评估”“必须发起重构收敛”“进入红线区”。
- 达到阈值后，后续改动必须优先评估拆分；超过重构或红线阈值时，不得继续无计划堆积复杂度。
- 行数阈值只是强触发信号；若复杂度已明显失控，应提前启动重构。

## 12. 小程序 UI 修改必须验证构建产物

- 以 `dist/build/mp-weixin` 生成产物为准，不以 Vue 源码为准。
- 每次 UI 改动后强制执行：检查 `src` → `npm run build:mp-weixin` → 微信开发者工具打开 `dist/build/mp-weixin`。
- 用户反馈“页面没变化”时，先判定运行态不是最新产物，不先猜样式。
- 顶部三层结构（导航层 / Hero 层 / Content 层）一次只改一层，禁止同时动多层。

## 13. 定期净化 steering

- 每周、发布前、重大 Spec 收尾后运行 `cd kaipai-frontend && npm run audit:steering`。
- 发现问题时，优先合并重复、迁移错层、归档历史、删除失效条目。

## 14. 微信小程序包体必须持续审计

- `mp-weixin` 不只看能不能 build，还必须持续检查主包 / 分包体积。
- 默认约束是单包不超过 `2 MB`；在接近阈值前就应启动分包评估，而不是等超限后再救火。
- 每次影响页面、静态资源、共享组件或大块脚本结构的改动后，应结合 `build:mp-weixin` 和 `audit:mp-package` 一起验证。
- 分包治理必须基于真实构建产物和 `app.json` 配置，不得凭主观印象判断包体归属。

## 15. 新功能模块默认先判断是否独立分包

- 后续开发不再默认把新功能直接放进主包。
- 新模块若具备独立入口、非 tab、可能持续增长资源或页面复杂度，应优先按独立分包设计。
- 只有启动即达、主链路必经、tab 直接依赖的能力，才允许默认放入主包。
